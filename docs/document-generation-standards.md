# DeliverMate 文档生成规范

## 目标

DeliverMate 生成的文档不应只是“把原始资料改写成更顺的文字”，而应成为可用于客户确认、内部评审、测试设计和交付落地的工作文档。

每类输出都应满足四个要求：

1. 可定位：读者能快速知道文档用途、适用对象和当前状态。
2. 可确认：范围、角色、规则、验收标准和待确认点都被显式写出。
3. 可追踪：需求、功能、测试和 UAT 之间能够建立编号和映射。
4. 可收敛：资料不足时明确标记缺口，不把猜测伪装成结论。

## 通用原则

### 1. 先讲范围，再讲功能

如果先列功能、后补边界，输出会越来越像“功能愿望清单”。每份需求文档必须先回答：

- 这份文档解决什么问题
- 哪些内容本轮明确纳入
- 哪些内容当前不承诺

### 2. 验收标准必须能执行

不要写成“体验良好”“操作方便”这类抽象表述。验收标准要能被客户、交付、测试三方复核。

### 3. 不确定内容必须沉淀到待确认区

原始资料没有明确说过的内容，不直接写成既定事实。统一进入：

- 待确认问题
- 假设与约束
- 暂不承诺范围

### 4. 文档要有角色视角

需求和功能不能只写“系统支持什么”，还要写清：

- 谁发起
- 谁使用
- 谁确认
- 谁承担交付动作

## 各类文档最低结构

### 需求文档

最低应包含：

- 文档定位
- 背景与问题
- 范围说明
- 目标与角色
- 需求明细
- 验收与完成定义
- 风险与待确认事项

每条需求建议包含：

- 需求 ID
- 需求名称
- 关联角色
- 优先级
- 需求说明
- 验收标准
- 待确认点

### 功能清单

功能清单不是需求文档的缩写版，而是“实现与确认之间的对齐表”。

最低建议列：

- 功能 ID
- 功能名称
- 用户价值
- 触发条件
- 关键输入
- 系统输出
- 业务规则
- 验收信号
- 待确认点

### 测试用例

最低应覆盖：

- 正常流程
- 资料缺失或异常流程
- 边界条件
- 权限或角色差异

### UAT 表

UAT 表必须偏向客户语言，而不是研发语言。至少包含：

- UAT ID
- 验收项目
- 业务场景
- 操作路径
- 验收标准
- 客户负责人
- 状态
- 备注

### 培训手册

最低应包含：

- 适用对象
- 培训目标
- 使用前准备
- 核心功能概览
- 推荐操作步骤
- 注意事项
- 常见问题

## 当前实现策略

当前项目使用“两阶段输出”：

1. 先让 AI 输出结构化 JSON。
2. 再由本地模板把 JSON 渲染成 Markdown 文档。

这样可以减少：

- 模型直接输出 Markdown 时的结构漂移
- 功能清单列不齐
- 验收标准和待确认点缺失
- 同一份资料多次生成时风格不稳定
